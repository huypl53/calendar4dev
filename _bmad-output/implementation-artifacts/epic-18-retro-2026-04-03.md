## What Didn't Go Well

### RFC 5545 Compliance — Repeated Off-by-One
- The DTEND exclusivity bug (RFC 5545: DTEND is exclusive for all-day events) appeared in three separate components: `day-header.tsx`, `week-header.tsx`, and the ICS export serializer
- Root cause: RFC 5545 compliance was handled locally in each component rather than through a shared utility
- Impact: All-day events would appear on one extra day in the UI and be exported/imported with off-by-one date ranges
- Fix applied: UTC date extraction + `date < endDate` comparison in all three locations

### Race Conditions — Systemic Pattern
Three separate race conditions appeared across the sprint, each independently discovered:

1. **Share token generation** (Epic 17): Two concurrent requests could both read `shareToken = null`, generate different tokens, and the first caller would receive a stale token. Fixed with conditional UPDATE (`WHERE share_token IS NULL`) + re-read.

2. **isReadOnly during loading** (Epic 11): While `sharedCalendars` was still loading, `sharedPermissions` was empty, so shared events briefly appeared editable. Fixed by defaulting to read-only when `sharedCalendars === undefined`.

3. **Password session invalidation** (Epic 11): After changing a password, existing sessions remained valid. Fixed by deleting all user sessions after password update.

- Root cause: No systematic "concurrent access" checklist in story acceptance criteria
- Impact: Low exploitability but would cause confusing UX or potential security concern

### Memory Management in Long-Lived Tabs
- **Notification service** (Epic 12): `notifiedIds` Set grew unbounded in long-running tabs — never pruned entries. Fixed by tracking fire times and pruning entries older than 5 minutes.
- **Toast timers** (Epic 2/deferred): `setTimeout` IDs were not tracked, so manually dismissed toasts still had running timers. Fixed with module-level `toastTimers` Map.
- Root cause: Long-lived tab behavior was not a test concern — only instantiated in unit tests, not in "runs for 8 hours" scenarios

### UX Polish Gaps
- "Today" button showed stale date if the app tab remained open past midnight
- Settings panel state persisted when mobile sidebar slid off-screen and re-opened
- Accent color checkmark used hardcoded `text-white` which has poor contrast on light accent presets
- These were all caught in code review and fixed, but indicate a pattern of not testing edge states

## Key Insights and Lessons Learned

1. **RFC 5545 requires a shared library** — Any story touching iCalendar format should use a centralized parsing/serialization utility. Inline date comparisons in UI components will drift from the spec.

2. **Concurrent access needs an explicit checklist** — Before implementing any write endpoint, explicitly ask: "What happens if two requests hit this simultaneously?" This question was not in any acceptance criteria across 18 epics.

3. **Long-lived tab behavior is an underserved test surface** — Memory growth, stale timestamps, timer accumulation, and midnight boundary crossings don't appear in unit tests. Need explicit test scenarios or review checklist items.

4. **The code review workflow is the quality gate** — It caught issues that unit tests missed, specifically state management timing bugs and security-adjacent concerns (Cache-Control, session invalidation). The workflow is worth the overhead.

5. **Boring architecture compounds** — React Query's `networkMode`, Drizzle's type safety, Hono's middleware stack — each boring choice paid dividends when the complexity arrived in later epics.

6. **"What NOT to Build" is a first-class deliverable** — The consistent discipline around deferring service workers, offline mutation queues, email notifications, and token expiry kept the codebase maintainable. This should continue in future epics.
